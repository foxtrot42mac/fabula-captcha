# The Layers We Built for Ourselves

## On the social nature of the technology stack, and what happens when machines stop needing it

---

There is one simple question worth asking yourself when you look at any technology layer: *who* does it exist for?

Not "why"—there's usually a ready-made answer for that in the documentation. Specifically *who*. Who consumes this format, protocol, language? Whose life does it make easier? And what happens if that consumer changes?

I want to trace a single thought—one not fully formed, one that makes no claim to manifesto status. Call it an observation from a practicing engineer about the direction our entire industry seems to be heading. Slowly, with enormous inertia, but heading nonetheless.

---

## The history of abstraction as the history of concealment

The entire history of programming is a history of progressively hiding details from people.

Machine code hid electrical signals. Assembly hid numeric opcodes. Compilers hid assembly. High-level languages hid memory management and system calls. Frameworks hid routine patterns. IDEs hid file structure, compilation, linking.

Every new layer did the same thing: it relieved people of the need to think about the layer beneath it. Not prohibited—specifically *relieved of the need*. You can still write assembly. It's just that in the overwhelming majority of cases, you don't have to.

Large language models are the next step in this sequence. And possibly a qualitatively different step, because they conceal not one specific layer but the very idea of layer-by-layer manual construction.

---

## Source code as a social contract

Here it's worth pausing to consider what source code actually is. Not in the technical sense—in the *social* sense.

The machine doesn't need source code. A processor executes machine instructions. It is indifferent to whether those instructions were written by a person in a text editor, generated by a compiler, or produced by any other means. Source code exists because *people* needed to read, understand, discuss, review, and coordinate work among themselves. It is a contract between developers, not a requirement of the machine.

Source code performs three functions simultaneously. First, it is a *specification*—a description of what the system should do. Second, it is a *translation input*—the data fed into a compiler or interpreter. Third, it is a *debugging artifact*—something to return to when things go wrong.

LLMs are already capable of taking over the first two functions: understanding intent and producing an executable result. The third function—debugging—deserves its own discussion. In the world of C++, you don't need source code at runtime for debugging. You need debug symbols—a separate artifact that maps machine addresses to human-readable names. By analogy, in a system where code is generated by a model, the debugging artifact is no longer the program's source text but the *captured intent*—what the person wanted to achieve, along with a trace of the decisions the model made.

This is a different artifact. Unfamiliar, not necessarily textual. But functionally sufficient.

## Ten layers, built by people for people

Let's walk through several key components of the modern stack and ask that same question: *for whom?*

**The web server.** This is a historical intermediary. It exists because someone needed to explain to one machine how to talk to another—and explain it in a format that people could read, agree on, and standardize. Configuration files, routes, middleware chains—all of it is a human interface to a fairly simple task: accept a byte stream, process it, return a byte stream. When the system controls both ends of the communication, the intermediary in human-readable form becomes optional.

**The browser.** In essence, a translator from byte streams to video memory. It takes HTML, CSS, JavaScript, images in standardized formats—and turns all of it into pixels on a screen. Each of those formats is the product of lengthy negotiations between people. W3C, RFC, ECMA—these are committees where *people* agreed on representations convenient for *people*. The machine doesn't need those agreements. A model capable of generating the final representation directly has no need for an intermediate human-readable layer.

**Network protocols.** TCP, HTTP, TLS—these are message brokers with rigorously specified formats. The specifications were written so that independent development teams could create compatible implementations. This is a coordination mechanism between people. When the same system controls both sides of a channel, the packet format can be anything at all—as long as it's efficient.

In every one of these cases, we see the same picture. The layer exists not because the machine requires that particular structure, but because *people* needed a way to coordinate. All ten, twenty, fifty layers of the modern stack are social infrastructure that we built for ourselves.

---

## What is changing

Machines are ceasing to need human guidance at every step. This doesn't mean all layers will vanish. It means they will transform.

An LLM can already generate binary firmware for a simple microcontroller. There is nothing fundamental preventing further progress—producing executable code without an intermediate source representation, constructing efficient binary protocols without deferring to RFCs, composing visual output without the cascade of HTML → CSS → rendering.

But—and this matters—the layers don't disappear without a trace. They transform from *mandatory stops* into a *library of patterns*. HTTP doesn't cease to exist; it ceases to be the only possible way to organize data exchange. It becomes one template among many that the system can use when appropriate (for instance, when interacting with legacy infrastructure) and ignore when it's superfluous. The model needs "rails"—architectural patterns, standard solutions, proven structures. But these are *internal* rails, under no obligation to be expressed in human-readable form.

One more detail about the models themselves is worth noting. The name "Large Language Model" carries a built-in limitation. Language is only one channel, one artifact of human perception. Internally, the model operates not on language but on logical patterns, statistical regularities, structures. Language is simply how you feed it input and receive output in a format humans understand. A more accurate name would be *Large Logic Model*. The abbreviation stays the same, but the meaning is broader: a system that operates on patterns, for which words are just one of many possible interfaces.

## Everything in between is the system's internal affair

An architectural picture begins to emerge that looks roughly like this:

*Human → thin translation layer (language, sound, pixels) → bits → ... → bits → thin translation layer → human.*

Everything represented by the ellipsis is the system's internal affair. We are accustomed to this "internal affair" being organized into dozens of layers with human-readable interfaces between them. But that was only the case because a person stood at every junction: a sysadmin, a network engineer, a front-end developer, a back-end developer, a DevOps engineer. Each of them needed a representation they could understand.

Remove people from the intermediate points, and the need for human-readable interfaces *between layers* falls away. What remains is the need for human-readable interfaces *at the boundaries*—where a person states an intent and receives a result.

Text, incidentally, is also an artifact. Inside the model there are no "words"—there are numerical representations. Tokens are a wrapper for humans. But this wrapper is *necessary*, because at one end of the system there is always a person who thinks in words, images, and sounds.

## Counterarguments that cannot be ignored

It would be dishonest to present this picture without acknowledging serious objections.

**Legacy doesn't surrender.** COBOL still runs banking systems. Fortran is alive and well in scientific computing. The transition from one paradigm to another in critical infrastructure takes not years but decades. Existing layers will persist for a very long time, and any new system will need to interoperate with them. This is not an objection to the direction—it is an objection to the speed.

**Verification and security.** How do you audit a system that has no readable intermediate representation? Current security practices—code review, static analysis, formal verification—are anchored to source code. Without it, we lose not merely convenience but an entire class of tools for ensuring correctness. Proof assistants like Coq or Lean operate on formal specifications at the source-code level. Without a readable intermediate layer, they have nothing to work with. This may be the most serious challenge: fundamentally new methods of verification will be required, and they do not yet exist.

**Nondeterminism.** The chain "source code → compiler → binary" is deterministic: the same input always yields the same output. The chain "intent → LLM → binary" is probabilistic. For consumer applications, this may be acceptable. For nuclear reactor control, medical devices, avionics—it is not. And it remains unclear whether this problem can be solved in principle, not merely in practice.

**Displacement of complexity, not its elimination.** There is a risk that the "rails"—the internal patterns and architectural templates the model will use—will reproduce the same complexity as existing layers, only in a different, less transparent form. What we get may not be simplification but *concealment*—which is fundamentally not the same thing.

These objections are real, and they will not resolve themselves. But they do not negate the observable direction—they define its boundaries and its pace.

---

## The only thing that remains

This entire edifice—layers, protocols, languages, formats, specifications—was built by people, for people. Not for machines. Machines patiently accepted our formats, executed our abstractions, obeyed our conventions—not because those conventions were optimal, but because machines could not propose an alternative. Now, it seems, they are beginning to.

If you follow this line to its conclusion—through all the caveats, counterarguments, decades of migration, and unsolved verification problems—one thing comes into view on the horizon: the entire intermediate infrastructure, created for coordination between people, is gradually becoming optional to the extent that people are leaving the intermediate points.

But at the boundaries of the system, people remain. A person formulates intent. A person perceives the result. A person decides whether the system understood what was wanted. The thin translation layer at input and output—the interface between human thought and machine computation—is not merely preserved. It becomes *the only layer worth optimizing*.

And here a paradox emerges that is worth stating plainly. For decades, we perfected everything that sits *between* people: protocols, architectures, data formats, layers of abstraction. We built ever more elaborate structures inside the system, operating on the assumption that infrastructure complexity *is* progress. But if the internal layers can now restructure themselves, then the only thing that remains fixed—and therefore the only thing that warrants human attention—is the point of contact with the person. Not how the machine thinks. But how the person understands.

The boundary of human perception—the speed at which we read; the depth at which we trust; the threshold beyond which complexity ceases to be transparent and becomes noise—this is the true scarce resource of any system. Not computational power, not bandwidth, not storage capacity. All of those scale. Human attention does not. A person's ability to distinguish in a split second a correct result from a plausible one does not. And that is precisely why the future of system design is not system design at all. It is the design of the last millimeter: the thin, nearly weightless membrane where machine computation becomes human understanding. Everything else, the machine will figure out on its own. That millimeter, it will not.
